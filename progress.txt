What is already materially implemented
1. Discovery / frontend marketplace foundation
- Home, header, footer, discovery hero, search box and event rails were upgraded.
- Event cards and event detail page were redesigned closer to marketplace patterns.
- Heavy seat-map routes are lazy-loaded.
- Search suggestions now have a real backend contract and respect publication state instead of reading every seed event blindly.

2. Seat-map architecture and performance
- Heavy geometry moved out of the main JS bundle into static JSON under public/maps.
- Municipal background artwork is now externalized as a standalone SVG asset under public/seatmaps.
- Municipal background export was sanitized to produce valid SVG markup for the external asset, avoiding broken rendering from HTML-only entities.
- Municipal background backdrop now preserves the original sector floors, stage artwork and legends while stripping seat hotspots and seat-area wrappers from the static asset, and is rendered in its own image layer instead of a nested SVG `<image>` tag.
- Runtime seat-map loader fetches geometry on demand.
- Seat interactions were optimized for heavy SVG maps.
- Event detail pages now rely on manifest totals instead of inline geometry, avoiding crashes before the heavy map loads.

3. Inventory / orders
- Temporary seat holds exist.
- Review reservations for corporate/manual flows exist.
- Runtime inventory hydration exists.
- Orders are persisted locally.
- Duplicate purchase of already sold seats is blocked.
- Order cancellation releases reserved inventory again.

4. Payments
- Payment domain now exists locally.
- Pix and card authorize locally.
- Corporate orders go to under_review.
- Under-review orders can be settled later.
- Payment refund exists for cancellation flow.
- Payment state is separated from order state.

5. Tickets / wallet
- Tickets are issued locally after approval.
- Tickets are linked to order, seat, section and account.
- Tickets can be cancelled.
- Wallet token and QR/barcode payload are generated locally.

6. Notifications / analytics
- Notification outbox exists locally.
- Order confirmation, ticket issued, under-review and cancellation notifications exist.
- Business analytics events exist for hold, payment, approval, issuance, notification and cancellation.
- Progress tracking now has a live execution checklist and this status file.

7. Operations / backoffice
- Backoffice local exists at /operacao.
- The /operacao route is now treated explicitly as platform administration, not as the organizer surface.
- Review queue for corporate/manual orders exists.
- Backoffice can approve under-review orders.
- Backoffice can deny under-review orders.
- Backoffice can cancel approved orders.
- Backoffice can inspect payments, tickets, notifications and analytics.
- Backoffice now exposes operational diagnostics with severity, affected references and suggested actions.
- Backoffice now exposes local runbooks for approval, cancellation and alert triage.
- Organizer "Meus eventos" now exists at /produtor/meus-eventos and /organizador/meus-eventos.
- The organizer view now consolidates catalog, orders, revenue, tickets, notifications and attention state per event.
- The organizer view now also exposes real publication and unpublish actions against the backend catalog.
- Header and organization modules now separate the organizer surface from the platform admin route instead of routing producer access straight to /operacao.

8. Account area
- Account dashboard shows orders.
- Account dashboard now filters orders by status and exposes order drilldown with items, payment and lifecycle logs.
- Account dashboard shows payments.
- Account dashboard shows issued tickets.
- Account dashboard now opens wallet-linked ticket details and surfaces local notification diagnostics for each order.

9. API boundary
- Local domain APIs now exist for catalog, inventory, orders, payments, tickets, notifications, analytics, organizer and backoffice.
- Event runtime, seat hold flow, checkout creation, account reads and backoffice reads/actions already consume this API layer.
- Contract tests now cover the backend-facing API boundary.

10. Developer tooling
- A local Gemini terminal helper is now wired as `npm run eni`.
- The helper now carrega configuracao de `.env` ou `.gemini`, ganhou arquivos de exemplo versionados, aceita prompt externo versionado, sobe em `gemini-3-flash-preview` por padrao, tem atalhos dedicados para Flash/Pro/auto-fallback, e expoe um atalho local para o Gemini CLI.

11. Real backend foundation
- Supabase/Postgres real now exists for the product instead of only browser-local persistence.
- A migration runner now versions the database schema under `server/db/migrations`.
- The real schema now covers profiles, venues, events, sessions, sections, seats, seat_holds, orders, order_items, payments, tickets, notifications, analytics_events and audit_log.
- A server-side catalog domain now exists with upsert flows for venue, event, session, section and seat data.
- A transactional ticketing domain now exists with server-side hold expiration, seat locking, order creation, payment orchestration, approval, cancellation, ticket issuance and notification creation.
- Backend smoke scripts now validate Supabase admin access, direct Postgres access and an end-to-end flow of hold -> under_review -> approve -> tickets plus direct pix -> cancel.

12. First HTTP/BFF slice
- A real HTTP backend now exists under `server/http` using the Supabase/Postgres domains instead of in-browser storage.
- The exposed surfaces now cover organizer snapshot/publication, catalog publication/search, runtime catalog hydration, holds, orders, account reads and the backoffice API.
- The frontend operations, organizer, inventory, checkout and account APIs now support remote-first fetching through `VITE_BACKEND_URL` with local fallback if the backend is unavailable.
- Home, PDP and buyer entrypoints now honor backend publication state instead of exposing every seed event unconditionally.
- A dedicated HTTP smoke test now boots the server on an ephemeral port and validates health, organizer snapshot/publication, buyer flow, account reads and `/api/backoffice`.

13. Producer CRUD and backend-backed public catalog
- The organizer module now creates, edits and archives real events in Postgres instead of only publishing/unpublishing seed snapshots.
- The organizer dashboard now exposes a real event editor dialog backed by HTTP routes for create, update, editor read and archive.
- New events created in the organizer module now flow into the public catalog through backend list/detail/runtime routes, instead of remaining invisible outside the dashboard.
- Catalog list, detail, runtime and search now read backend-created published events directly, with local seeds used only as bootstrap/templates when needed.
- The HTTP smoke suite now validates producer CRUD plus public catalog visibility for created events.
- The producer dashboard now behaves more like a cockpit than a list view, with revenue and payout charts, agenda and preview blocks, a detailed per-event table and an explicit payout information card.
- The producer module now prioritizes financial clarity with gross, fee, payout-ready and occupancy visibility in the main surface instead of scattering those signals across isolated cards only.

14. Support and self-service foundation
- A real support domain now exists in Postgres with `support_cases` migration, persistence and audit logging.
- The HTTP backend now exposes support-case creation plus account-scoped support-case listing.
- The account area now exposes a support surface with operational FAQs, self-service trails and a form to open support requests tied to the account and, optionally, to an order.
- Local and remote API contracts now cover support-case creation and account history, and the HTTP smoke suite validates support case creation end-to-end.

15. Remote-first account authentication and cross-device history
- A dedicated customer-account backend now exists in Postgres with remote password registration, login, profile update and logout activity logging.
- The HTTP backend now exposes `register`, `login`, `logout` and account profile read/update routes for the buyer account surface.
- The React auth provider now operates in remote-first mode, persisting the returned account/session locally only as a client cache and falling back to the old local auth flow if the backend is unavailable.
- The account area now shows recent account activity and profile updates synced from the backend when available.
- Orders, payments, tickets, notifications and support cases can now be revisited across devices as long as the buyer signs into the same remote-backed account.
