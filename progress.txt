What is already materially implemented
1. Discovery / frontend marketplace foundation
- Home, header, footer, discovery hero, search box and event rails were upgraded.
- Event cards and event detail page were redesigned closer to marketplace patterns.
- Heavy seat-map routes are lazy-loaded.

2. Seat-map architecture and performance
- Heavy geometry moved out of the main JS bundle into static JSON under public/maps.
- Municipal background artwork is now externalized as a standalone SVG asset under public/seatmaps.
- Municipal background export was sanitized to produce valid SVG markup for the external asset, avoiding broken rendering from HTML-only entities.
- Municipal background backdrop now preserves the original sector floors, stage artwork and legends while stripping seat hotspots and seat-area wrappers from the static asset, and is rendered in its own image layer instead of a nested SVG `<image>` tag.
- Runtime seat-map loader fetches geometry on demand.
- Seat interactions were optimized for heavy SVG maps.
- Event detail pages now rely on manifest totals instead of inline geometry, avoiding crashes before the heavy map loads.

3. Inventory / orders
- Temporary seat holds exist.
- Review reservations for corporate/manual flows exist.
- Runtime inventory hydration exists.
- Orders are persisted locally.
- Duplicate purchase of already sold seats is blocked.
- Order cancellation releases reserved inventory again.

4. Payments
- Payment domain now exists locally.
- Pix and card authorize locally.
- Corporate orders go to under_review.
- Under-review orders can be settled later.
- Payment refund exists for cancellation flow.
- Payment state is separated from order state.

5. Tickets / wallet
- Tickets are issued locally after approval.
- Tickets are linked to order, seat, section and account.
- Tickets can be cancelled.
- Wallet token and QR/barcode payload are generated locally.

6. Notifications / analytics
- Notification outbox exists locally.
- Order confirmation, ticket issued, under-review and cancellation notifications exist.
- Business analytics events exist for hold, payment, approval, issuance, notification and cancellation.
- Progress tracking now has a live execution checklist and this status file.

7. Operations / backoffice
- Backoffice local exists at /operacao.
- Review queue for corporate/manual orders exists.
- Backoffice can approve under-review orders.
- Backoffice can deny under-review orders.
- Backoffice can cancel approved orders.
- Backoffice can inspect payments, tickets, notifications and analytics.

8. Account area
- Account dashboard shows orders.
- Account dashboard shows payments.
- Account dashboard shows issued tickets.

9. API boundary
- Local domain APIs now exist for catalog, inventory, orders, payments, tickets, notifications, analytics and backoffice.
- Event runtime, seat hold flow, checkout creation, account reads and backoffice reads/actions already consume this API layer.
- Contract tests now cover the backend-facing API boundary.
